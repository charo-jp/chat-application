## About the project
Web Chat Application.

## Commands

### Backend (`/backend`)
```bash
npm start          # Run dev server (tsx, port 8080)
npm run build      # TypeScript compilation
npm run test       # Vitest test suite
npm run coverage   # Test coverage report
npm run seed       # Seed the database via Prisma
npx prisma migrate dev   # Run DB migrations
npx prisma studio        # Open Prisma Studio GUI
```

### Frontend (`/frontend`)
```bash
npm run dev        # Vite dev server
npm run build      # TypeScript check + Vite production build
npm run lint       # ESLint
npm run storybook  # Storybook on port 6006
npm run start:all  # Run dev + storybook in parallel
```

## Architecture

This is a monorepo with a separate `frontend/` and `backend/` directory — no shared packages or root-level scripts tie them together.

### Backend — Fastify + Prisma

- **Entry point**: `backend/index.ts` — creates the Fastify server, registers plugins, mounts routes, and sets a global error handler.
- **Routes**: `backend/routes/` — each file exports a Fastify plugin. Routes are mounted with a prefix in `index.ts`.
- **Schemas**: `backend/schemas/` — holds Prisma `select` objects used to shape API responses (avoid over-fetching). Keep response shape definitions here, not inline in route handlers.
- **Database**: PostgreSQL via Prisma ORM. The Prisma client is generated into `backend/generated/prisma/`.
- **Tests**: `backend/__test__/routes/` — Vitest, configured in `backend/vite.config.ts` to match `**/*.test.ts` in `__test__/`.
- **Env**: `backend/.env` sets `DATABASE_URL` pointing to a local PostgreSQL database named `howdi`, user `chat_app_dev`.

### Frontend — React 19 + TanStack Router + Vite

- **Routing**: TanStack Router with file-based routing under `frontend/src/routes/`. The Vite plugin auto-generates route types — adding a new file to `routes/` creates a new page.
- **UI components**: ShadCN primitives built on Radix UI, stored in `frontend/src/components/ui/`. Add new primitives with the ShadCN CLI; don't hand-edit the generated files.
- **Styling**: Tailwind CSS v4, configured via the Vite plugin (no separate `tailwind.config.js`).
- **Path alias**: `@/` maps to `frontend/src/` — use this for all non-relative imports.
- **State**: Zustand is installed but not yet wired up. Auth state will live there.
- **Storybook**: Stories live alongside components in `frontend/src/stories/`. Vitest is integrated into Storybook.

### Data flow

The frontend calls the Fastify REST API on port 8080. Authentication is not yet implemented — routes that require auth are marked with `// TODO: requires auth` in `backend/routes/users.ts`.

## Backend coding conventions

### File layout

Each route file has two clearly separated sections marked with comment dividers:

```ts
// API Definitions------------------------------------------------------
// ... handler functions

// Routes Definitions------------------------------------------------------
// ... route plugin
```

### Handlers

- Exported as named `const` arrow functions, typed explicitly:
  ```ts
  export const loginHandler = async (
    request: FastifyRequest<{ Body: LoginRequest }>,
    reply: FastifyReply,
  ): Promise<void> => { ... }
  ```
- Naming: `{verb}{Entity}Handler` — e.g. `loginHandler`, `getMyUserInfoHandler`.
- First line of every handler logs the call:
  ```ts
  request.log.info({ userId: id }, "getMyUserInfoHandler called")
  ```
  Use a structured object `{ key: value }` for IDs, not an interpolated string.
- Log again when the resource is found: `request.log.info({ userId: user.id }, "handler found user")`.
- On early error exit: log with `request.log.error(...)`, then `reply.status(code).send({ error: message })` and `return`.

### Route plugins

- Exported as named `async function {domain}Routes(server: FastifyInstance)`.
- Auth guard applied with a `preHandler` hook, not per-route: `server.addHook("preHandler", server.authenticate)`.

### Error responses

Always send `{ error: "message" }` — never bare strings or nested objects.

### Prisma queries

- Always pass a `select` object — never fetch the full model.
- Define `select` objects in `schemas/` (named `{scope}Select`) and derive the response type with `Prisma.ModelGetPayload`:
  ```ts
  export const myInfoSelect = { id: true, username: true } as const;
  export type MyUserInfoResponse = Prisma.UserGetPayload<{ select: typeof myInfoSelect }>;
  ```

### Config and constants

- All magic strings and numbers live in `config.ts` as named `const` exports in `SCREAMING_SNAKE_CASE`.
- Constant-object values use `as const` for literal-type narrowing (e.g. `TOKEN_TYPES`).

### Imports

- Always include the `.ts` extension in local imports.
- Use the `type` keyword for type-only imports: `import type { FastifyInstance } from "fastify"`.

### Plugins

- Wrap with `fastify-plugin` (`fp`) so decorations are visible across the server scope.
- Extend `FastifyInstance` / `FastifyRequest` via `declare module` in the plugin file, not elsewhere.

### Tests

- **Unit tests** call the handler function directly using `createMockRequest` / `createMockReply` from `__test__/helpers.ts`.
- **Validation tests** must go through Fastify's schema validation step — boot a minimal app with `app.inject()` instead of calling the handler directly.
- Structure: one happy-path test at the top of the `describe` block, then failure/edge cases in numbered nested `describe` blocks:
  ```ts
  describe("loginHandler", () => {
    it("happy path ...", async () => { ... });

    describe("1. User Existence", () => { ... });
    describe("2. Password Validation", () => { ... });
  });
  ```
- Mock at module scope with `vi.mock(...)`, clear in `beforeEach(() => vi.clearAllMocks())`.
- Name mock-arrangement helpers `arrange{Scenario}` (e.g. `arrangeSuccessfulLogin`).
